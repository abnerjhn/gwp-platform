Especificaciones de Desarrollo: Plataforma de Gestión Integrada GWP/NDC1. Contexto y ProblemaEstamos gestionando una consultoría compleja compuesta por 3 especialistas (Finanzas, Legal, Coordinación) que tienen contratos individuales (TdR) pero deben trabajar de forma integrada.Problema: Los TdR individuales tienen cronogramas desconectados y lógicas inversas (piden diseño antes que diagnóstico).Solución: Esta aplicación web (Streamlit + Supabase) debe actuar como un "Corrector de Flujo", forzando la colaboración mediante "Compuestas Lógicas" (Gates) y dependencias cruzadas.2. Stack Tecnológico RequeridoFrontend: Streamlit (Python).Backend/DB: Supabase (PostgreSQL).Almacenamiento: Supabase Storage (para evidencias PDF/Word).Visualización: Plotly (Gantt) y Graphviz/Mermaid (Flujos).3. Arquitectura de "Doble Entrada"La aplicación debe manejar dos realidades simultáneas:Vista Operativa (Realidad Técnica): Se trabaja por "Fases" y "Paquetes Integrados" (ej. Taller de Co-Creación).Vista Contractual (Pagos): Se reporta por "Productos TdR" (ej. Producto 2.1, Producto 3.2).Nota: Una actividad operativa (ej. "Taller") alimenta el progreso de múltiples productos contractuales a la vez.4. Roles de UsuarioFINANZAS (Patricio): Genera propuestas de mecanismos y datos.LEGAL (Constanza): Valida viabilidad normativa (Semáforo Legal).COORD/QA (Astrid): Valida inclusión/género y gestiona hitos de pago.GOBIERNO (Visor): CONAF/MMA (solo visualizan avances y validan hitos finales).5. Lógica de "Gates" (Crucial)El sistema debe impedir el avance a la siguiente fase si no se cumplen los requisitos de la actual:GATE 1 (Fin Mes 1): No se permite crear fichas de "Mecanismos" si la "Línea Base Mínima" no está al 100%.GATE 2 (Fin Mes 2.5): No se permite pasar a "Diseño Operativo" (Manuales/GIS) si el mecanismo no tiene Doble Check (Legal + Género).
